# Epic Prompts — 2026-07-07

## Как лучше работать

Рекомендация:

- текущий чат использовать для стратегических решений, аудита, приоритизации и handoff;
- каждый крупный epic выполнять в новом диалоге;
- внутри одного эпика можно держать весь цикл: анализ -> реализация -> тесты -> итог;
- следующий epic лучше запускать в новом чате, чтобы не тащить лишний контекст и не размывать фокус.

Почему это лучше:

- меньше контекстного шума;
- проще держать одну цель на один диалог;
- ниже риск, что старые решения случайно перетянут неактуальные допущения;
- удобнее фиксировать итог по каждому этапу.

Исключение:

- если epic маленький или прямо зависит от только что сделанных локальных изменений, можно продолжать в текущем чате.

## Общий шаблон для любого epic

Используй этот каркас, если захочешь запускать дополнительные подпроекты:

```text
Работаем в репозитории CareerSignal HH.

Сначала изучи текущую реализацию в коде и документации, не делай предположений по памяти.
Нужен полный цикл работы по epic: аудит текущего состояния -> проектирование минимально правильного решения -> реализация -> тесты -> краткий итог с рисками и следующим шагом.

Требования:
- не ломать существующий online-first сценарий;
- не трогать без необходимости уже рабочие auth, UI и search flows;
- все изменения делать прагматично, без лишней архитектурной теории;
- если видишь устаревшую документацию, обновляй её по ходу;
- если есть техдолг, закрывай только тот, который реально мешает этому epic;
- после изменений прогони релевантные тесты и сообщи результат;
- если есть спорные решения, принимай их самостоятельно и фиксируй в docs.
```

## Tail-0

```text
Работаем в репозитории CareerSignal HH.

Нужно выполнить Tail-0 stabilization batch перед следующей волной разработки.

Цель:
- выровнять документацию с фактическим состоянием проекта;
- зафиксировать baseline online-first режима;
- убрать хвосты, которые мешают корректно планировать следующие эпики, но не уходить в бесконечную полировку.

Сделай:
1. Проведи короткий аудит docs и README на предмет устаревших предпосылок.
2. Проверь, что текущий roadmap не противоречит реальному коду.
3. Обнови документацию там, где она говорит, что UI отсутствует, manual user OAuth не реализован или live HH режим ещё недоступен.
4. Зафиксируй baseline commands для рабочего online сценария.
5. Найди оставшиеся неархитектурные хвосты и раздели их на:
- блокирующие;
- неблокирующие.
6. Закрой всё блокирующее, что можно закрыть в рамках этой сессии.
7. Прогони релевантные тесты/проверки и дай краткий итог.

Важно:
- не переписывай roadmap заново без необходимости;
- не начинай новые фичи;
- не трогай крупные доменные куски, если это не нужно для стабилизации.
```

## Epic A — Candidate Profile + Search Positioning

```text
Работаем в репозитории CareerSignal HH.

Нужно реализовать Epic A: Candidate Profile + Search Positioning.

- обновить позиционирование кандидата;
- привести search presets к реальному целевому рынку;

Сначала:
1. Изучи текущие candidate/profile/presets/scoring/search-related файлы.
2. Сверь это с актуальным ТЗ и текущей стратегией проекта: CRM, Bitrix24, integrations, AI automation, no-code, analyst roles.

3. Улучши include/exclude логику, если это нужно.
4. При необходимости подстрой scoring/search-lab так, чтобы он лучше различал целевые и шумовые вакансии.

Результат должен дать:
- более чистую очередь вакансий;
- понятную матрицу целевых ролей;
- воспроизводимую логику поиска.

После реализации:
- прогони релевантные тесты;
- если тестов не хватает, добавь их;
- кратко покажи, как теперь запускать и проверять новые пресеты.
```

## Epic B — Letter Engine Rewrite

```text
Работаем в репозитории CareerSignal HH.

- убрать шаблонность и слабые generic drafts;

4. Если нужно, добавь режим явной диагностики причин reject.
5. Обнови docs/examples.

Требования:
- не делать LLM-зависимый дизайн;
- логика должна быть прозрачной и тестируемой;
- слабые письма должны отсеиваться автоматически;
- хорошие письма должны быть короче, конкретнее и привязаны к вакансии.

После реализации:
- добавь тесты на хорошие и плохие кейсы;
- прогони релевантный тестовый набор;
- покажи итоговые правила качества письма.
```
